# Roadmap

## Принципы

* Сквозное шифрование (E2EE): сервер не получает ключи расшифровки контента.
* Ники обязательны и уникальны; регистрации доступны только по invite-ключам.
* Собираются только метаданные, необходимые для доставки. Они не считаются
  зашифрованными и требуют отдельной минимизации.
* Криптографические протоколы не реализуются самостоятельно: используются
  проверенные библиотеки Signal Protocol и MLS.
* Транспорт первого этапа — обычный HTTPS/JSON. Контент внутри него —
  непрозрачный Signal-контейнер, а не самодельная криптография.

* [x] Отдельный CLI для ручной проверки transport API.
* [ ] PostgreSQL, миграции и наблюдаемость для production.
* [ ] Интеграционные тесты API и ограничение частоты запросов.
* [ ] Утвердить криптографический профиль: PQXDH + Double Ratchet, safety
  numbers/QR-верификация и политика ротации prekeys.

## Этап 1 — личные текстовые диалоги

* [x] Каталог публичных pre-key пакетов и одноразовых ключей для Signal Protocol.
* [x] Интеграция поддерживаемой libsignal, без собственной реализации примитивов.
* [x] Локальная зашифрованная история Android-клиента.
* [x] Создание Signal-сессии и отправка серверу только шифротекста.
* [x] Foreground long-poll, системные уведомления без текста и локальные счётчики непрочитанных.
* [x] Подтверждения: одна галочка после приёма сервером, две после открытия
  диалога получателем.
* [x] Числовой safety number для сверки identity-ключей вне мессенджера.
* [x] QR-представление safety number и проверка QR из выбранного изображения.
* [x] QR-проверка через системную камеру без постоянного camera-permission.
* [ ] Live QR-сканер и отслеживание смены ключа.
* [x] Очистка диалога локально или у обеих сторон: удаление серверного
  шифротекста и маркер синхронизации для другого клиента.
* [ ] Production push wake-up.
* Push без текста и иных конфиденциальных данных.

## Этап 2 — вложения

* [x] Клиентское AES-256-GCM шифрование картинок и войсов отдельным случайным ключом.
* [x] Отдельно зашифрованные превью и очищенный полноразмерный JPEG.
* [x] Ключ и метаданные вложения передаются внутри Signal E2EE-сообщения.
* [x] Ограничения размера, квота, проверка формата и явное удаление blob.
* [x] Полноэкранный просмотр изображения в Android без plaintext-файла.
* [x] Конфигурационный storage seam: SQLite по умолчанию, S3-заглушка
  fail-closed до готовности bucket policy и интеграции.
* [ ] Вынести blob из SQLite в закрытый S3/MinIO bucket через storage abstraction.
* [ ] Политика retention, сборка мусора для осиротевших blob и ротация бэкапов.

## Этап 3 — группы

* Группы и роли администратора.
* MLS для группового E2EE и обновления ключей при смене состава.
* Выбрать зрелую MLS-библиотеку и пройти отдельное тестирование совместимости.
* Зашифрованные сообщения, картинки, войсы и удаление в группах.

## Этап 4 — десктоп и эксплуатация

* Привязка устройства через QR и явное подтверждение с основного устройства.
* Десктоп-клиент на Kotlin Compose Multiplatform, синхронизация ключей и
  история в локальном шифрованном виде. Разработчику нужен JDK; релизные
  пакеты включают runtime и не требуют установленной Java у семьи.
* Подписанный канал обновлений: проверка ключа подписи и безопасный перезапуск;
  на Android установка остаётся подтверждаемым системным действием.
* Версионированный Hiddi Envelope для realtime/WebSocket и desktop: явные
  типы кадров, версия, лимиты и совместимость. Он не заменяет Signal и не
  передаёт открытый текст; до этой потребности используем HTTPS.
* Внешний криптографический аудит, threat-model review, резервное копирование
  сервера и процедура реакции на инциденты.
