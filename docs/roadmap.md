# Roadmap

## Принципы

* Сквозное шифрование (E2EE): сервер не получает ключи расшифровки контента.
* Ники обязательны и уникальны; регистрации доступны только по invite-ключам.
* Собираются только метаданные, необходимые для доставки. Они не считаются
  зашифрованными и требуют отдельной минимизации.
* Криптографические протоколы не реализуются самостоятельно: используются
  проверенные библиотеки Signal Protocol и MLS.
* Транспорт первого этапа — обычный HTTPS/JSON. Контент внутри него —
  непрозрачный Signal-контейнер, а не самодельная криптография.

## Ближайшая очередь реализации

1. Стабилизация, единая Android design system и закрытый production-релиз
   для первых приглашённых пользователей.
2. Перенос локальной истории из единого encrypted JSON в постраничное
   зашифрованное хранилище без полного чтения файла.
3. Версионированный Direct Envelope, ответы и редактирование сообщений.
4. Пересылка с повторным шифрованием текста и вложений.
5. Emoji-панель и E2EE-реакции в личных чатах и группах.
6. PIN/биометрия, multi-device/desktop и production-инфраструктура.

## Этап 1 — личные текстовые диалоги

* [x] Каталог публичных pre-key пакетов и одноразовых ключей для Signal Protocol.
* [x] Интеграция поддерживаемой libsignal, без собственной реализации примитивов.
* [x] Локальная зашифрованная история Android-клиента.
* [x] Создание Signal-сессии и отправка серверу только шифротекста.
* [x] Foreground long-poll, системные уведомления без текста и локальные счётчики непрочитанных.
* [x] Подтверждения: одна галочка после приёма сервером, две после открытия
  диалога получателем.
* [x] Числовой safety number для сверки identity-ключей вне мессенджера.
* [x] QR-представление safety number и проверка QR из выбранного изображения.
* [x] QR-проверка через системную камеру без постоянного camera-permission.
* [x] Предупреждение о смене ранее подтверждённого identity-ключа и блокировка
  отправки до повторной сверки.
* [ ] Live QR-сканер.
* [x] Очистка диалога локально или у обеих сторон: удаление серверного
  шифротекста и маркер синхронизации для другого клиента.
* [x] Удаление отдельного своего сообщения локально или у обеих сторон:
  двухступенчатое контекстное меню, удаление шифротекста и надёжный ACK-маркер
  второму клиенту.
* [x] Игнор-лист личных диалогов: сервер молча отбрасывает новые сообщения и
  вложения заблокированного отправителя, не раскрывая ему факт блокировки.
* [x] Отдельный Android-экран основных настроек: аватар, видимое имя,
  неизменяемый `@nickname`, описание и явное объяснение границы E2EE.
* [x] Просмотр чужого профиля из поиска, списка диалогов и шапки чата:
  аватар, имя, ник, описание и переход в личный диалог.
* [x] Приватные локальные контакты: добавление и удаление из профиля, отдельный
  раздел по кнопке с возвратом к диалогам и Keystore-зашифрованное хранение без
  серверного social graph.
* [x] Ленивая история личного диалога в UI: первая страница последних 50
  сообщений и подгрузка предыдущих страниц при прокрутке вверх с сохранением
  позиции. Compose не держит весь диалог одновременно.
* [ ] Перенести локальную историю из единого Keystore-зашифрованного JSON в
  постраничную БД с шифрованием отдельных записей, чтобы чтение страницы не
  требовало кратковременной расшифровки всего локального файла.
* [ ] E2EE-синхронизация контактов между своими устройствами; сервер получает
  только ciphertext backup и не видит состав адресной книги.
* [ ] Редактирование своих сообщений как отдельное версионированное E2EE-событие
  с явной пометкой «изменено» и защитой от повторов.
* [ ] Версионированный Hiddi Direct Envelope внутри Signal ciphertext:
  тип события, случайный message ID, версия протокола и строгие лимиты. Это
  общий фундамент для ответов, редактирования, реакций и пересылки; сервер
  по-прежнему получает только непрозрачный Signal-контейнер.
* [ ] Ответ на сообщение: зашифрованная ссылка на message ID и короткая
  локальная цитата, без серверной связи между сообщениями.
* [ ] Пересылка личных и групповых сообщений в другой личный диалог или группу.
  Клиент создаёт новое E2EE-сообщение без серверной ссылки на исходный чат.
  Текст шифруется заново, а вложение получает новые ключи и новый ciphertext;
  имя первоначального автора передаётся только по явному выбору пользователя.
* [ ] Unicode emoji-панель в composer с компактным избранным набором и
  системным полным выбором emoji.
* [ ] E2EE-реакции на сообщения: add/remove-события с emoji, target message ID,
  аутентифицированным автором и защитой от повторов. Сервер не видит emoji.
* [ ] Production push wake-up.
* Push без текста и иных конфиденциальных данных.

## Этап 3 — группы

* [x] Архитектура групп, модель ролей и fail-closed MLS transport-контракт.
* [x] Выбран OpenMLS (RFC 9420) как общий Rust core для Android и desktop.
* [x] Выделен `group-mls-core`: OpenMLS 0.8.1, MLS 1.0 и versioned opaque
  envelopes с unit-тестами.
* [x] Android Keystore storage boundary для непрозрачного MLS group state.
* [x] Воспроизводимая Android arm64 native-сборка OpenMLS core через NDK.
* [x] Минимальный JNI bridge Android → Rust для fail-closed проверки MLS envelope.
* [x] MLS round-trip test: создание, Welcome, join и application message между
  двумя независимыми client state, включая восстановление группы из OpenMLS
  `StorageProvider`.
* [x] Core-провайдер persistent MLS storage: OpenMLS SQLite provider с
  AES-256-SIV шифрованием каждой записи и тестом восстановления группы.
* [x] Android Keystore → JNI: 64-байтный ключ MLS storage создаётся локально,
  зашифрованно хранится вне backup и передаётся Rust только в память процесса.
* [x] Локальное создание и восстановление одноучастниковой MLS-группы в
  persistent provider; без UI и сетевой отправки до Commit/Welcome transport.
* [x] Welcome/join/application-message round-trip между двумя независимыми
  зашифрованными SQLite MLS-профилями.
* [x] Server transport: routing metadata, роли и адресная доставка только
  versioned opaque MLS envelopes без расшифровки на сервере.
* [x] Rust bridge OpenMLS для Android и desktop: persistent signer, KeyPackage,
  Add Commit, Welcome/join и двусторонние application messages.
* [x] Надёжный server inbox/ACK и idempotency-ID для повторной отправки
  непрозрачных MLS events без дубликатов.
* [x] Android: Keystore-зашифрованные group directory/history/outbox,
  двухпользовательская группа, экран переписки и фоновый live-приём.
* [x] Terminal: общий Rust/OpenMLS core, зашифрованное состояние и команды
  `group publish/create/list/send/inbox/watch/sync`.
* [x] Добавление третьего участника владельцем: одноразовый KeyPackage,
  OpenMLS Add Commit существующим участникам и Welcome новому участнику.
* [x] Запас одноразовых MLS KeyPackage: Android и terminal автоматически
  публикуют не меньше 10 пакетов, сервер хранит до 32 и атомарно расходует
  только один на приглашение.
* [x] Выход с устройства: отзыв серверного токена и подтверждаемое локальное
  удаление Signal/MLS-ключей, истории, контактов и вложений.
* [x] Права владельца: локальная очистка истории и надёжное удаление группы
  у всех участников через server marker/ACK.
* [x] Роли owner/admin/member, назначение администраторов владельцем, удаление
  участника через OpenMLS Remove Commit и обязательная новая epoch.
* [x] Явное создание группы из главного меню: E2EE-название и первый участник
  из локальных контактов либо по `@nickname`. Внутри группы участники
  приглашаются теми же двумя способами.
* [ ] Кросс-платформенные тесты MLS и отдельный криптографический review.
* [x] Живой кросс-клиентский тест Android + два независимых terminal-профиля:
  приглашение третьего участника, обмен до удаления, Remove Commit и отсутствие
  маршрутизации удалённому профилю после смены epoch.
* [x] Зашифрованные текстовые сообщения в первом двухпользовательском group UI.
* [x] Удаление своего группового сообщения локально или у всех: versioned
  delete payload внутри OpenMLS и физическое удаление исходного шифротекста.
* [x] Add/Remove Commit и Welcome объединены с encrypted crash-atomic operation
  journal в одной native SQLite-транзакции. Android и terminal сначала
  идемпотентно переносят journal в свои durable outbox, затем отправляют ACK.
* [x] Зашифрованные картинки и войсы в группах: один серверный ciphertext с
  проверкой текущего членства, ключ и описание только внутри MLS application
  message, отдельное превью и полноэкранный просмотр.
* [ ] Cursor-pagination старых MLS application events и ленивая подгрузка
  истории группы без повторного применения Commit/Welcome.
* [ ] Ответы, пересылка и реакции в MLS-группах через версионированные
  application payload. Удалённый участник не получает события новой epoch.

## Этап 4 — десктоп и эксплуатация

* Привязка устройства через QR и явное подтверждение с основного устройства.
* Десктоп-клиент на Kotlin Compose Multiplatform, синхронизация ключей и
  история в локальном шифрованном виде. Разработчику нужен JDK; релизные
  пакеты включают runtime и не требуют установленной Java у семьи.
* Подписанный канал обновлений: проверка ключа подписи и безопасный перезапуск;
  на Android установка остаётся подтверждаемым системным действием.
* Версионированный Hiddi Envelope для realtime/WebSocket и desktop: явные
  типы кадров, версия, лимиты и совместимость. Он не заменяет Signal и не
  передаёт открытый текст; до этой потребности используем HTTPS.
* Единая кросс-платформенная модель Direct/MLS payload для текста, вложений,
  ответов, пересылок, редактирования и реакций; одинаковые test vectors для
  Android, terminal и desktop.
* Внешний криптографический аудит, threat-model review, резервное копирование
  сервера и процедура реакции на инциденты.
