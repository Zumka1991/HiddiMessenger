# Архитектура

## Компоненты

```text
Android / desktop client  -- HTTPS + WebSocket -->  Rust API  --> PostgreSQL
        |                                             |
        |-- локальная зашифрованная БД                 +--> object storage
        |                                                     (ciphertext only)
        +-- Signal / MLS: ключи и расшифровка только на устройстве
```

Rust API отвечает за приглашения, учётные записи, поиск по никнейму, каталог
публичных ключей, маршрутизацию шифротекстов и срок их хранения. Метаданные и
ссылки на blob хранятся в основной БД; сами blob доступны через отдельную
границу object storage. В production основная БД — PostgreSQL, object storage —
S3-совместимое хранилище; SQLite служит локальным backend ранней стадии.

## Развёртывание

Сервер поставляется в Docker-образе и запускается через Docker Compose. Том
данных монтируется отдельно от контейнера. API публикуется только на localhost,
а TLS, HSTS, лимиты тела запроса и rate limit на внешнем контуре выполняет
reverse-proxy. До production SQLite будет заменён PostgreSQL, а backend
зашифрованных вложений — S3/MinIO. Имена объектов случайны, bucket закрыт от
публичного чтения, доступ выдаёт только авторизованный Rust API.

## Идентификация и доступ

У пользователя есть обязательный уникальный ник (`@nickname`) и одно или
несколько устройств. Регистрация возможна лишь по invite-ключу. Сервер хранит
только хеш invite-ключа и помечает одноразовый ключ использованным атомарно.

Устройство создаёт ключевую пару локально. При регистрации в API передаётся
только публичный identity key. Дальнейшая аутентификация должна быть
challenge-response подписью этим ключом; временный bearer-токен в текущем
каркасе — переходный механизм, не финальный дизайн.

## Данные и шифрование

| Данные | Где и в каком виде |
|---|---|
| Ник, идентификатор учётной записи, публичные ключи | Сервер, открыто |
| Текст, emoji, реакции | Только клиент / серверный шифротекст |
| Войсы и изображения | Клиентский шифротекст: сейчас SQLite BLOB, затем S3-compatible object storage |
| Ключи вложений | В E2EE-сообщении, не на сервере |
| Метаданные доставки | Сервер; минимизируются и имеют retention policy |

## Криптографический профиль

«Самое сильное» — не один алгоритм, а проверенная композиция протоколов,
правильное управление ключами и независимый аудит. Для личных диалогов выбран
Signal Protocol: PQXDH для асинхронного старта с post-quantum KEM, затем Double
Ratchet для уникального ключа каждого сообщения, прямой секретности и
post-compromise security. При реализации используется поддерживаемая библиотека
libsignal, а не собственный Rust/Kotlin-код криптографии.

Для групп выбран стандартизованный MLS (RFC 9420) через зрелую реализацию. Для
каждой картинки или войса клиент генерирует отдельный 256-битный случайный ключ
и шифрует данные AES-256-GCM; ключ вложения передаётся только внутри
E2EE-сообщения. Изображение состоит из двух независимо зашифрованных объектов:
очищенного полноразмерного JPEG и компактного превью. Явная связь объектов
передаётся только участникам в Signal-сообщении; сервер всё ещё может пытаться
сопоставить загрузки по времени, размеру и адресатам. Associated data связывает
шифротекст с версией протокола, случайным binding ID, видом и MIME-типом.

Приватные identity/prekey/ratchet-ключи остаются на устройстве в Android Keystore
и зашифрованной локальной БД. Для локальной блокировки приложения применяется
PIN с Argon2id, а не парольная «производная» для ключей сообщений. Сеть — только
TLS 1.3; резервные копии никогда не содержат приватные ключи или открытый текст.

PQXDH повышает защиту от сценария «записать сейчас, расшифровать позже» при
пассивном квантовом противнике, но не решает задачу аутентификации против
активного квантового противника. Поэтому QR-сверка safety numbers остаётся
обязательной функцией интерфейса. Нельзя заменять эти протоколы собственными
AES-схемами или хранить приватные ключи на сервере.

## Удаление и сроки хранения

Сервер удаляет шифротекст сообщения и blob вложения по запросу «удалить у
всех», если он ещё хранится. Клиентам направляется E2EE-команда удаления.
Удаление не может отменить уже скачанную локальную копию, экспорт или скриншот
у другого участника — это должно быть явно отражено в интерфейсе.

Подтверждение доставки не удаляет серверный шифротекст: история хранится до
явного удаления или настроенной retention policy. Это не даёт новому устройству
автоматически прочитать историю — для этого потребуется клиентский master key и
безопасная привязка устройства по QR. Бэкапы сервера шифруются и имеют
документированный срок ротации; удаление отражается в последующей ротации.

## Границы угроз

Проект защищает содержимое от сервера, провайдера хранения и сетевого
наблюдателя при корректной TLS-настройке. Он не скрывает от сервера факт,
время, размер и адресатов передачи, а также никнеймы. Компрометация устройства
или разблокированный телефон остаются отдельными угрозами: нужны Android
Keystore, защита приложения и безопасная привязка устройств.
