# Группы: MLS-архитектура

Этот документ описывает обязательный контракт до появления первого group endpoint.
Группа не будет реализована как общий статический AES-ключ или набор личных
сообщений: оба варианта не дают корректной ротации ключей при исключении участника.

## Выбранный криптографический core

Клиентский MLS-core будет построен на OpenMLS (Rust, RFC 9420) и подключён к
Android/desktop через небольшой Rust bridge. Сервер Rust не расшифровывает MLS
состояние, application messages, commits или welcomes.

Первый crate `group-mls-core` уже закрепляет OpenMLS 0.8.1, MLS 1.0 и
версионированный opaque envelope. Он ещё не создаёт группы: это намеренная
граница, пока Android bridge и персистентное MLS state не пройдут тесты.

Android уже имеет `EncryptedGroupMlsState`: это Keystore-зашифрованное хранилище
непрозрачных байтов MLS state. Kotlin не интерпретирует эти байты; после bridge
их создаёт и проверяет только Rust/OpenMLS.

## Что видит сервер

* Случайный `group_id`, аккаунты текущих участников и их роли.
* Размер и время доставки непрозрачных blob.
* Порядковый номер server event, необходимый только для надёжной доставки.

Название, аватар, описание, MLS epoch, список добавлений/исключений и содержимое
сообщений находятся в зашифрованном клиентском group state. В будущем членство
можно будет дополнительно скрыть, но это не входит в первый семейный релиз.

## Обязательные MLS-события

1. Создатель получает key packages приглашённых устройств и локально создаёт MLS
   группу.
2. Для каждого приглашённого отправляется непрозрачный MLS Welcome; сервер не
   может прочитать его или сгенерировать сам.
3. Изменение состава оформляется MLS Commit. Клиент применяет commit только после
   криптографической проверки MLS и сохраняет новое состояние атомарно.
4. После удаления участника оставшиеся клиенты переходят в новую epoch. Удалённый
   участник не должен расшифровывать сообщения новой epoch.
5. Application messages принимаются клиентом только для известной группы и
   валидной epoch. Неизвестный/устаревший blob остаётся диагностическим событием,
   а не отображаемым текстом.

## Роли и политика первого релиза

* `owner` — создаёт группу, назначает/снимает администраторов, удаляет группу.
* `admin` — приглашает и удаляет участников.
* `member` — отправляет сообщения и добровольно выходит.
* Все изменения ролей и состава требуют MLS Commit; серверная роль — только
  авторизационный фильтр, не источник криптографической истины.

## Transport, который будет реализован

`POST /v1/groups` и последующие endpoints принимают только versioned opaque
envelopes: `welcome`, `commit` или `application`. Поля plaintext не появятся в
API. До подключения OpenMLS Android-клиент не будет показывать создание групп,
чтобы не выпускать небезопасную «заглушку группы».

## Проверка до релиза

* Кросс-платформенные векторы OpenMLS: Android ↔ desktop.
* Тест: добавление/удаление участника, новая epoch и невозможность расшифровки
  сообщения удалённым клиентом.
* Тесты повтора/перестановки group events и восстановления после перезапуска.
* Отдельный криптографический review перед семейным rollout.
