# Группы: MLS-архитектура

Этот документ описывает реализованный первый MLS-контур и обязательные условия
перед расширением состава групп.
Группа не будет реализована как общий статический AES-ключ или набор личных
сообщений: оба варианта не дают корректной ротации ключей при исключении участника.

## Выбранный криптографический core

Клиентский MLS-core построен на OpenMLS (Rust, RFC 9420) и подключён к
Android/desktop через небольшой Rust bridge. Сервер Rust не расшифровывает MLS
состояние, application messages, commits или welcomes.

Crate `group-mls-core` закрепляет OpenMLS 0.8.1, MLS 1.0 и версионированный
opaque envelope. Он создаёт persistent группу, хранит signer, выпускает
KeyPackage, Add Commit/Welcome, принимает Welcome/Commit и шифрует application
messages. Тот же cdylib вызывают Android и terminal/будущий desktop.

`group-mls-core` проверяет настоящий MLS round-trip: Alice создаёт группу,
добавляет Bob через Welcome, затем добавляет Charlie новым Commit/Welcome.
Bob применяет Commit, Charlie присоединяется, и оба расшифровывают сообщение
новой epoch. Затем Alice исключает Charlie настоящим OpenMLS Remove Commit:
Bob читает сообщение следующей epoch, а Charlie расшифровать его не может.
Тест также восстанавливает группу из `StorageProvider`: ratchet state не
сериализуется Hiddi вручную.

Android хранит Rust/OpenMLS SQLite в `noBackupFilesDir`; каждая запись
зашифрована AES-256-SIV ключом из Android Keystore. Отдельные Keystore-хранилища
держат group directory, локальную историю и retry-outbox. Kotlin не
интерпретирует приватное MLS state.

В Rust уже добавлен постоянный `EncryptedSqliteMlsProvider`: это upstream
OpenMLS SQLite `StorageProvider`, а не собственный формат состояния Hiddi.
Каждая сериализованная OpenMLS запись шифруется AES-256-SIV до записи в SQLite;
это необходимо, потому что часть ключей участвует в SQLite lookup и должна быть
детерминированной. Ключ длиной 64 байта передаётся только в память процесса
после получения из Android Keystore (на desktop — из OS keychain). JNI-вызов,
который создаёт/извлекает этот ключ, уже добавлен и вызывается при старте
Android-приложения. Вызов idempotent для того же ключа и fail-closed для другого.
Provider подключён к lifecycle Android и terminal-клиента. Сетевой повтор
защищён idempotency-ID и ACK после локального сохранения.

Core уже умеет создать и сохранить локальную одноучастниковую MLS-группу с
credential зарегистрированного `device_id`; этот идентификатор уже известен
серверу как технический идентификатор устройства и не является никнеймом или
названием группы. Android имеет UI MLS-группы: создание из поиска, приглашение
новых участников владельцем, список групп, текстовую переписку, локальную
очистку владельцем, надёжное удаление группы и фоновый приём.

Помимо unit-теста выполнен живой прогон Android и двух независимых
terminal-профилей: третий участник принял Welcome, существующий участник —
Commit, после чего сообщения новой epoch расшифровались всеми тремя. До
семейного релиза ещё нужен crash-atomic journal для узкого окна между OpenMLS
mutation и записью события в отдельный outbox.

Отдельный живой прогон проверил исключение: третий профиль получил сообщение
до Remove Commit, применил предназначенный ему Commit и удалил локальное MLS
state. Следующее сообщение новой epoch сервер ему уже не маршрутизировал.

## Нативная сборка Android

Для разработки bridge нужен Android NDK и Rust target `aarch64-linux-android`.
Сборка воспроизводится так:

```bash
export ANDROID_NDK_HOME="$HOME/Android/Sdk/ndk/29.0.14206865"
bash scripts/build-android-mls.sh
```

Она производит `libhiddi_group_mls_core.so` для `arm64-v8a`. JNI принимает
только `ByteArray` и проверяет versioned opaque envelope в Rust. Библиотека
упаковывается в APK и используется первым двухпользовательским group UI.

## Что видит сервер

* Случайный `group_id`, аккаунты текущих участников и их роли.
* Размер и время доставки непрозрачных blob.
* Порядковый номер server event, необходимый только для надёжной доставки.

Название, аватар, описание, MLS epoch, список добавлений/исключений и содержимое
сообщений находятся в зашифрованном клиентском group state. В будущем членство
можно будет дополнительно скрыть, но это не входит в первый семейный релиз.

## Обязательные MLS-события

1. Создатель получает key packages приглашённых устройств и локально создаёт MLS
   группу.
2. Для каждого приглашённого отправляется непрозрачный MLS Welcome; сервер не
   может прочитать его или сгенерировать сам.
3. Изменение состава оформляется MLS Commit. Клиент применяет commit только после
   криптографической проверки MLS и сохраняет новое состояние атомарно.
4. После удаления участника оставшиеся клиенты переходят в новую epoch. Удалённый
   участник не должен расшифровывать сообщения новой epoch.
5. Application messages принимаются клиентом только для известной группы и
   валидной epoch. Неизвестный/устаревший blob остаётся диагностическим событием,
   а не отображаемым текстом.

## Роли и политика первого релиза

* `owner` — создаёт группу, назначает/снимает администраторов, удаляет группу.
* `admin` — приглашает и удаляет участников.
* `member` — отправляет сообщения и добровольно выходит.
* Изменение состава требует MLS Commit и смены epoch. Роль меняет только
  полномочия управления и не даёт нового доступа к контенту, поэтому отдельный
  MLS Commit для смены owner/admin/member не требуется.
* Владелец может удалить admin/member; администратор — только member. Владелец
  не может быть удалён или понижен.

## Реализованный transport

`POST /v1/groups` создаёт только routing metadata: случайный group id,
участников и их роли. `POST /v1/groups/{group_id}/events` принимает только
versioned opaque envelopes: `welcome`, `commit` или `application`, и адресатов
из текущего server-side списка участников. Поля plaintext не появятся в API.
Для Welcome/Commit сервер разрешает отправку только owner/admin, но не считает
себя источником криптографической истины. Inbox использует long-poll, явный ACK
после локального сохранения и client event ID для безопасного повтора.
Добавление участника регистрируется отдельной идемпотентной outbox-операцией
до отправки Commit/Welcome. Исключение атомарно связывает OpenMLS Remove Commit
с удалением routing-членства; удаляемый получает помеченный Commit и стирает
локальное MLS state. Удаление группы создаёт для каждого участника надёжный
маркер, который ACK-ается только после удаления локального MLS state.

Текст и удаление сообщения имеют версионированный application payload внутри
OpenMLS. Delete payload содержит случайный message ID и может удалить только
сообщение того же аутентифицированного MLS-отправителя. После надёжной отправки
delete payload сервер физически удаляет адресные копии исходного ciphertext по
этому ID; содержимое команды и текста серверу недоступно.

## Проверка до релиза

* Кросс-платформенные векторы OpenMLS: Android ↔ desktop.
* Тест: добавление/удаление участника, новая epoch и невозможность расшифровки
  сообщения удалённым клиентом.
* Тесты повтора/перестановки group events и восстановления после перезапуска.
* Отдельный криптографический review перед семейным rollout.
